# 100oku Desire｜担当整理とファイル別タスク割り振り

## 1. 担当ごとの基本役割

### A: フロントエンド実装担当

**役割**

* React 実装全般
* 画面遷移の制御
* 状態管理
* アニメーション実装
* 全体の結合・動作保証

**責任範囲**

* アプリが最後まで通して動く状態を作る
* 画面同士をつなぐ
* データ、状態、UI を結合する
* GitHub Pages で動く状態にする

**基本方針**

* ロジックの最終実装責任者
* `App.tsx` / `hooks/` / `utils/` は A 主導
* B と C の変更を安全に取り込む

---

### B: 企画・コンテンツ担当

**役割**

* カード文言作成
* 診断ロジック設計
* ビジネス仮説整理
* 発表時の説明内容作成

**責任範囲**

* アプリの中身（何を選ばせるか）を決める
* 診断タイプの意味づけを決める
* 「なぜこのアプリに価値があるか」を言語化する
* 実装担当が迷わないよう、仕様を文章で固定する

**基本方針**

* B は**仕様を決める人**
* 直接コードを触るなら、**文言・データ系ファイルのみ**
* ロジックの実装自体は A が行うが、判定ルールの設計は B が持つ

---

### C: デモ・仕上げ担当

**役割**

* UI微調整
* 分析画面（デモ用）作成
* デモ動画撮影
* 最終確認・バグチェック

**責任範囲**

* 見た目を整える
* デモ映えする導線を作る
* 発表で使う画面・スクショ・動画を仕上げる
* 最終的な体験品質を上げる

**基本方針**

* C は**伝わり方を整える人**
* 直接コードを触るなら、**UI系ファイル中心**
* ロジックは壊さず、見た目・文言・分析デモを整える

---

## 2. 作業ルール（衝突防止）

### 原則

* **A = ロジックを持つ**
* **B = 仕様と文言を持つ**
* **C = 見た目とデモ品質を持つ**

### 触るファイルのルール

* `app/`, `hooks/`, `utils/` は **A 主担当**
* `data/`, `constants/copy.ts` は **B 主担当**
* `components/`, `screens/`, `styles/` は **C 主担当**
* `App.tsx` は **A だけが直接編集**
* B/C が仕様変更したい場合は、A に反映依頼 or PRベースで調整

### レビューの流れ

* B が文言・診断ルールを決める
* A が実装する
* C が見え方を調整する
* 最後に 3人でデモ動線確認

---

## 4. 進め方（おすすめ順）

### フェーズ1: 仕様固定

* **B** がカード内容・診断タイプ・文言を決める
* **A** が型を固める
* **C** が画面ラフを決める

### フェーズ2: 基盤実装

* **A** が `types/`, `hooks/`, `utils/`, `app/` を作る
* **C** が `components/`, `screens/`, `styles/` を作る
* **B** が `data/`, `constants/copy.ts` を埋める

### フェーズ3: 接続

* **A** が画面にロジックを接続
* **C** が見た目調整
* **B** が文言最終調整

### フェーズ4: デモ仕上げ

* **C** が分析画面・動画・スクショ
* **B** が発表内容
* **A** がバグ修正・公開確認

---

## 5. 最重要ルール

* **A は「動く責任」を持つ**
* **B は「中身の責任」を持つ**
* **C は「伝わる責任」を持つ**

この役割を崩さないと、短期開発でもかなり進めやすくなる。
